# Work Share Server Design Memory

Last updated: 2026-03-11 (SQLite Docker volume confirmed)

## Context
We are revising Work Share server design goals around user accounts, SSH keys, team organization, and team-scoped patch sharing.

## Confirmed Requirements (from user)
- Anyone may create a user account.
- The server stores each user's username, full name, email, password (hashed), and SSH public keys.
- A user may register multiple SSH keys (e.g. one per device).
- Each SSH key entry has a user-supplied label (e.g. "Work MacBook") for identification.
- SSH keys were originally intended for authenticating patch uploads from the plugin.
- SSH key authentication from the plugin is ON HOLD: the plugin will use username/password instead.
- VS Code provides `vscode.ExtensionContext.secrets` (SecretStorage) backed by the OS keychain
  (macOS Keychain, Windows Credential Manager, libsecret on Linux) — safe for storing credentials.
- The plugin will store username/password credentials using SecretStorage.
- After login, the server issues a short-lived session token; the plugin stores the token in SecretStorage and sends it on subsequent requests.
- Session tokens expire after 1 hour.
- The server supports silent token refresh so users are not prompted to re-enter credentials on expiry.
- Raw credentials are never sent on every request.
- During account creation, the user supplies a mutable password.
- The password will later be used to authenticate the user.
- Password storage must follow security best practices.
- Email addresses are globally unique.
- Team owners may add members by either username or email.
- Team owners may only add users that already exist in the system.
- Team invitations must be accepted by the target user before membership becomes active.
- Users should control patch sharing so their patches are not shared outside their teams.
- Both the plugin and website should restrict visibility of patches and activity to items generated by teammates.
- All shared teams are enabled for sharing by default.
- Users can enable or disable sharing with specific teams.
- Information streams should be filterable by team.
- Disabling sharing for a team pauses all outgoing information streams for that team.
- Team members should be able to see the sharing status of all members of their team.
- Historical information may remain visible on the website even after team sharing is disabled.
- Historical patch information should not be sent to the plugin when it is considered stale.
- A patch is considered stale when the same user has submitted a newer patch for the same file.
- When a user disables sharing with a team, the server records a timestamp of that event.
  - Patches uploaded before the disable timestamp remain visible to teammates.
  - Patches uploaded after the disable timestamp are hidden from teammates.
- Users can belong to more than one team.
- Any user may create one or more teams.
- Teams are identified by team name.
- Team names are globally unique.
- Team owner may delete the team.
- Any user may leave a team they are associated with.
- Initially, only the team creator can add other users to their team.
- Patches are shared only with users who share a common team.
- UI is needed for users to provide SSH key.
- Server interfaces are needed for organizing users into teams.

- The website uses a simple server-side session (HttpOnly cookie) — no complex token flow needed for the web UI.
